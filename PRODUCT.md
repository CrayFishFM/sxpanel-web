# Product

## Register

brand

## Platform

web

## Users

Two overlapping buyers land here: existing **txAdmin server owners** deciding whether to migrate, and **new FiveM/RedM server owners** picking an admin panel from scratch. Both are evaluating, not yet using — the page has to convince an owner to make an install decision, not walk an admin through a workflow. A secondary audience of day-to-day admins/staff cares more about what the panel lets them do once it's installed, but they're rarely the one deciding to install it.

## Product Purpose

sxPanel is a full-featured web panel and in-game menu for FiveM/RedM servers — a complete, drop-in overhaul of txAdmin with live console, deep insights, player management, granular permissions, and Discord integration. This site exists to convert a skeptical server owner into a downloader: prove the switch is safe, show the feature depth, and get them to install. Success is a completed download of the latest release, with Discord/notify-list signup as the fallback for visitors not ready to commit yet.

## Positioning

Switching from txAdmin to sxPanel is a zero-risk upgrade, not a rewrite: existing servers, databases, and configs work unmodified, while the panel underneath is dramatically more capable.

## Conversion & proof

- Primary CTA: **Download the latest release** (GitHub Releases). Secondary fallback: **Join Discord** or the **notify list**, for visitors not ready to commit yet.
- The line a visitor remembers after 10 seconds: *"Swap the folder, keep everything, get a lot more."*
- Belief ladder: (1) this is fully compatible with what I already run — no data migration, no reconfiguration → (2) it does meaningfully more than txAdmin — live analytics, in-game admin menu, Discord bot, 40+ permissions → (3) it's built for serious, real communities, not a hobby side project → (4) I can be running it in minutes, so the risk of trying it is near zero.
- Proof on hand: none yet — sxPanel is early-stage with no public testimonials, server counts, or logos to lean on. Persuasion should rest on the compatibility story and concrete feature/spec detail (real command names, real numbers) rather than social proof; don't fabricate placeholder stats or logos to fill the gap.

## Brand Personality

Technical and precise, but approachable — not intimidating. The voice speaks in real terminology and real numbers (40+ permissions, 96h insights history, <60s deploy) rather than marketing abstractions, while staying welcoming to less technical server owners rather than gatekeeping around jargon.

## Anti-references

Avoid the generic SaaS-template default this category slides into: soft cream/gradient backgrounds, the hero-metric-card cliché, tiny uppercase eyebrows stacked above every section, glassmorphism used decoratively. The existing violet-accented neutral surface already breaks from that mold — new work should keep pushing away from it, not drift back toward it.

## Design Principles

- **Compatibility-first proof.** Every claim in service of "drop-in, zero risk" should be shown concretely — real migration commands, real folder names — not asserted abstractly.
- **Technical precision over marketing fluff.** Reach for the actual number, command, or feature name already in the product before reaching for a generic adjective.
- **Approachable without dumbing down.** The audience is technical, but copy and layout should never require insider knowledge to follow on a first pass.
- **Show, don't just tell.** Product mockups, realistic panel data, and real Discord embed replicas do more work than icon grids and abstract illustrations.
- **Earn attention outside the SaaS playbook.** Deliberately avoid the cream/gradient/eyebrow scaffolding that saturates this category; the brand's own visual language should carry the page.

## Accessibility & Inclusion

Standard WCAG AA baseline: solid text contrast, full keyboard navigation, and a working `prefers-reduced-motion` alternative for every animation. No unusual accommodations beyond that baseline.
